# Plan: Merge Release-Orchestrator Deploy Ideas

## Goal

Create one working java-client release/deploy flow that combines the current repo workflow with the useful parts of the issue-577 feature branch, without using `git merge`.

The resulting workflow must be triggerable from the `zenbpm` release orchestrator via `repository_dispatch`, update client API/proto sources from the matching backend tag, keep Maven versions with the leading `v`, commit/tag/release the java-client repo, and publish artifacts to Maven Central Portal.

## Confirmed Decisions

- Trigger from orchestrator: `repository_dispatch` with type `release_published`.
- Dispatch payload field: `github.event.client_payload.release_tag`.
- Maven artifact versions keep the leading `v`, for example `v1.4.0`.
- Maven coordinates stay as-is: `org.zenbpm`.
- Backend sources are downloaded by `gh api` from `pbinitiative/zenbpm` at the release tag:
  - `openapi/api.yaml`
  - `pkg/zenclient/proto/zenbpm.proto`
- Maven Central publishing uses Central Portal, not legacy OSSRH/Nexus.

## Recommended Changes

### 1. `.github/workflows/release.yaml`

Update the existing workflow instead of replacing it wholesale.

- Add `repository_dispatch`:
  - `types: [release_published]`
- Keep `workflow_dispatch` for manual fallback/testing, but normalize both trigger paths into one `RELEASE_TAG` value.
- Use `github.event.client_payload.release_tag` for orchestrator runs.
- Update concurrency to use the normalized release tag.
- Add validation that `RELEASE_TAG` exists and starts with `v`.
- Keep the GitHub App token step because the workflow needs access to both `zenbpm-java-client` and `zenbpm`.
- Keep `fetch-depth: 0` because the workflow checks tags and creates a new tag.
- Keep the existing `gh api` download approach for both `api.yaml` and `zenbpm.proto`; do not clone the backend repo and do not update only `api.yaml`.
- Keep `mise run set-release-version` so Maven versions remain exactly `vX.Y.Z`.
- Prefer `mvn -B clean verify -Dgpg.skip=true` or keep `mise run build` for the normal build; do not sign during this build step.
- Add GPG import/setup before deploy using secrets:
  - `MAVEN_GPG_PRIVATE_KEY`
  - `MAVEN_GPG_PASSPHRASE`
- Add Maven Central Portal deploy step:
  - `mvn -B deploy -P release -DskipTests -Dgpg.skip=false --settings .github/settings.xml`
- Keep committing the changed POMs, `api.yaml`, and `zenbpm.proto`.
- Keep tagging and pushing `HEAD:main` plus the release tag.
- Keep creating the GitHub Release and uploading built jars.

Suggested release order:

1. Resolve and validate `RELEASE_TAG`.
2. Create GitHub App token.
3. Checkout java client.
4. Fail if java-client tag already exists.
5. Setup mise tools.
6. Download backend `api.yaml` and `zenbpm.proto` from the backend tag.
7. Set Maven version to `RELEASE_TAG`.
8. Build/verify unsigned.
9. Commit release changes.
10. Import GPG key.
11. Deploy to Maven Central Portal.
12. Tag and push commit/tag.
13. Create GitHub Release.
14. Upload build artifacts.

Note: this preserves the current behavior of pushing the release commit to `main`. If the Central deploy succeeds but push/tag later fails, the run must be manually recovered because the version may already exist in Central.

### 2. `.github/settings.xml`

Add Maven settings for Central Portal credentials, based on the feature branch:

```xml
<settings xmlns="http://maven.apache.org/SETTINGS/1.0.0"
          xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
          xsi:schemaLocation="http://maven.apache.org/SETTINGS/1.0.0
                              https://maven.apache.org/xsd/settings-1.0.0.xsd">
    <servers>
        <server>
            <id>central</id>
            <username>${env.MAVEN_CENTRAL_USERNAME}</username>
            <password>${env.MAVEN_CENTRAL_TOKEN}</password>
        </server>
    </servers>
</settings>
```

Add release/build plugins from the feature branch, adjusted to preserve current coordinates:

## Explicitly Excluded From The Feature Branch

- Do not strip the leading `v` from Maven versions.
- Do not change `org.zenbpm` Maven coordinates to `org.pbinitiative.zenbpm`.
- Do not rename Java packages to `org.pbinitiative.*`.
- Do not clone `pbinitiative/zenbpm` just to copy `api.yaml`.
- Do not update only `api.yaml`; update both `api.yaml` and `zenbpm.proto`.
- Do not include the large API YAML content changes from the feature branch manually; the release workflow should pull sources from the backend tag.

## Verification

Local checks before opening/running the workflow:

```bash
mise trust
mise install
mvn -B clean verify -Dgpg.skip=true
```

Version preservation check:

```bash
RELEASE_TAG=v1.4.0 mise run set-release-version
mvn help:evaluate -Dexpression=project.version -q -DforceStdout
```

Expected version output:

```text
v1.4.0
```

Central-profile dry build without publishing:

```bash
mvn -B clean verify -P release -Dgpg.skip=true
```

Workflow syntax/contract checks:

- Confirm `release.yaml` references `github.event.client_payload.release_tag` for `repository_dispatch`.
- Confirm manual fallback still accepts a version input if kept.
- Confirm source downloads use these backend paths:
  - `/repos/pbinitiative/zenbpm/contents/openapi/api.yaml?ref=${RELEASE_TAG}`
  - `/repos/pbinitiative/zenbpm/contents/pkg/zenclient/proto/zenbpm.proto?ref=${RELEASE_TAG}`

End-to-end GitHub test from a safe branch/tag:

```bash
gh api repos/pbinitiative/zenbpm-java-client/dispatches \
  -f event_type=release_published \
  -f client_payload='{"release_tag":"vX.Y.Z"}'
```

After a successful run, verify:

- `main` contains a release commit updating POM versions, `api.yaml`, and `zenbpm.proto`.
- Java-client tag `vX.Y.Z` exists.
- GitHub Release `vX.Y.Z` exists and has jars attached.
- Maven Central Portal has published:
  - `org.zenbpm:zenbpm-java-client:vX.Y.Z`
  - `org.zenbpm:zenbpm-client-core:vX.Y.Z`
  - `org.zenbpm:zenbpm-spring-boot-starter:vX.Y.Z`
